## Scope and design intent

Accounts Payable (AP) in FOS ERP follows the same pattern as AR (Phase 3): one **financial document** per supplier bill, a separate **AP subledger open-item** table, and atomic posting through the central posting engine (Phase 2B).

```text
MasterVendor (soft ref) ──► VendorInvoice (financial document)
                                ├── VendorInvoiceLine[]
                                ├── VendorInvoiceSourceLink[] (optional)
                                └── PayableOpenItem (1:1 via sourceVendorInvoiceId, Phase 4A4)
```

Workflow and posting detail: [`AP_WORKFLOW.md`](AP_WORKFLOW.md). Calculation semantics: [`AP_CALCULATION_RULES.md`](AP_CALCULATION_RULES.md). Status: [`AP_STATUS.md`](AP_STATUS.md).

## One financial document: `VendorInvoice`

A single `VendorInvoice` row represents the **financial** supplier bill regardless of how goods or services were received operationally.

| `VendorInvoiceType` | Typical use |
|---------------------|-------------|
| `GOODS` | Stock / material purchase |
| `SERVICE` | Subcontractor, professional services |
| `EXPENSE` | Overheads, utilities, travel |
| `ASSET` | Capital purchase |
| `MIXED` | Combined bill |

**Physical receipt ≠ financial invoice.** A goods receipt (GRN) or purchase receipt is an operational/logistics event. The vendor invoice is the finance document that recognises liability, input tax, and expense/asset debits. When Purchase backend exists, `VendorInvoiceSourceLink` can reference a GRN or PO by id — there is **no FK** to deferred Purchase models.

Direct entry (no source links) is fully supported.

---

## Vendor identity: reuse `MasterVendor`


- `VendorInvoice.vendorId` and `PayableOpenItem.vendorId` store `MasterVendor.id` as a **soft reference** (no Prisma FK, no `FinanceVendor` table).
- Header snapshots (`vendorCodeSnapshot`, `vendorNameSnapshot`, GSTIN/PAN/state/address) are persisted at document time for audit and reporting.
- Deleting a `MasterVendor` row does **not** block or cascade to existing invoices (verified in foundation tests).

## Numbering: three distinct identifiers

| Field | Purpose | Phase 4A1 state |
|-------|---------|-----------------|
| `draftReference` | Tenant-scoped internal draft key (`@@unique([tenantId, draftReference])`) | Required on create; generated by repository helper |
| `supplierInvoiceNumber` | Supplier's bill number as printed on the tax invoice | Required; stored with normalised copy |
| `vendorInvoiceNumber` | FOS internal posted document number | **Nullable until Phase 4A4 posting** (`@@unique([tenantId, legalEntityId, vendorInvoiceNumber])`) |

Do not conflate supplier number with FOS number. Posted numbering will consume `FinanceNumberSeries` with `documentType = VENDOR_INVOICE` (enum extended in migration; series row creatable but not consumed in 4A1).

## Supplier invoice normalisation and duplicate prevention

**Normalisation** (`vendor-invoice-number-normalization.ts`):

- Trim, uppercase, collapse internal whitespace.
- Preserves `/` and `-` — `ABC-001` and `ABC/001` remain distinct.

**Uniqueness key** (claimed in Phase 4A3, not enforced in 4A1):

```text
supplierInvoiceUniquenessKey =
  tenantId|legalEntityId|vendorId|financialYearId|supplierInvoiceNumberNormalized
```

- Column is nullable in 4A1 so parallel early drafts with the same supplier number are allowed.
- When set, `@unique` on `supplierInvoiceUniquenessKey` prevents duplicate supplier bills for the same vendor + FY + legal entity.

## Amount semantics (documented for 4A2+; not calculated in 4A1)

All money fields use **`Decimal(18, 4)`** (base-currency mirrors prefixed with `base*`). **Amounts are always positive**; direction is never encoded with negative decimals.

| Field | Meaning |
|-------|---------|
| `grossAmount` | Total before discount and tax |
| `discountAmount` | Commercial discount |
| `taxableAmount` | Base for GST |
| `inputCgstAmount` / `inputSgstAmount` / `inputIgstAmount` / `inputCessAmount` | Input tax components |
| `otherRecoverableTaxAmount` / `nonRecoverableTaxAmount` | Non-GST recoverable / absorbed tax |
| `freightAmount` / `otherChargeAmount` / `roundOffAmount` | Charges and rounding |
| `invoiceGrandTotal` | Final supplier invoice value **before** TDS settlement effect |
| `tdsBaseAmount` / `tdsAmount` | TDS base and withheld amount |
| `vendorPayableAmount` | Vendor liability for future posting (may differ from grand total when TDS at invoice) |

**Phase 4A2 (shipped)** implements the calculation/validation engine that computes these fields — see [`AP_CALCULATION_RULES.md`](AP_CALCULATION_RULES.md) for GST/ITC/RCM/TDS/currency/duplicate/account-resolution rules and accounting-preview invariants. The engine is pure/DB-optional (`calculateVendorInvoiceSync` / `calculateVendorInvoice`) and does **not** write to these columns itself — persistence still happens in the (future) draft workflow service (Phase 4A3). 4A1 rows still store zeros on new drafts until that write path exists.

## GST / TDS / ITC foundation fields

| Concern | Header fields | Notes |
|---------|---------------|-------|
| GST treatment | `taxTreatment` (`VendorInvoiceTaxTreatment`) | Purchase-side enum — distinct from AR `SalesInvoiceTaxTreatment` |
| ITC eligibility | `itcEligibility` (`InputTaxCreditEligibility`) | Accounting review state, not statutory determination |
| TDS recognition | `tdsRecognitionMode` (`NOT_APPLICABLE`, `AT_INVOICE`, `AT_PAYMENT`) | Policy placeholder for 4A2+ |
| TDS detail | `tdsSectionCode`, `tdsSectionDescription`, `tdsRate`, `tdsBaseAmount`, `tdsAmount` | |
| GL account placeholders | `vendorPayableAccountId`, `inputCgstAccountId`, … `tdsPayableAccountId`, `roundOffAccountId` | Optional FKs → `accounts`; populated before post in later phases |
| Line-level | `hsnSacCode`, per-component GST rates/amounts, optional `taxTreatment` / `itcEligibility` overrides | |

Place of supply and company/vendor GSTIN snapshots support future GST reporting; no return filing in 4A1.

## Lines: `VendorInvoiceLine`

- One-to-many with **Cascade** delete from invoice.
- `lineType`: `ITEM`, `SERVICE`, `EXPENSE`, `ASSET`, `FREIGHT`, `OTHER_CHARGE`.
- Optional soft `itemId` (MasterItem), `debitAccountId`, `costCentreId`.
- Optional per-line source document pointers (`sourceLinkType`, `sourceDocumentId`, …) for 3-way match UX later.
- `@@unique([vendorInvoiceId, lineNumber])`.

---

## Optional source links: `VendorInvoiceSourceLink`

Soft links to upstream documents without Purchase FKs:

| `VendorInvoiceSourceLinkType` | Intended source |
|-------------------------------|-----------------|
| `PURCHASE_ORDER` | PO id (future) |
| `GOODS_RECEIPT` | GRN id (future) |
| `PURCHASE_RECEIPT` | Non-stock receipt |
| `CONTRACT` / `PROJECT` / `OTHER` | Reference ids |

- **Cascade** delete with invoice.
- `@@unique([vendorInvoiceId, sourceType, sourceDocumentId])`.

## AP subledger: `PayableOpenItem`

Separate table from `ReceivableOpenItem` — no shared polymorphic subledger.

| `PayableOpenItemSide` | Meaning |
|-----------------------|---------|
| `CREDIT` | Vendor invoice liability (amount owed **to** vendor) |
| `DEBIT` | Payment, advance, debit note, credit adjustment (future) |

| `PayableOpenItemDocumentType` | Future document |
|--------------------------------|-----------------|
| `VENDOR_INVOICE` | Posted vendor invoice |
| `VENDOR_PAYMENT` | Outgoing payment |
| `VENDOR_ADVANCE` | Advance to vendor |
| `VENDOR_DEBIT_NOTE` / `VENDOR_CREDIT_ADJUSTMENT` | Adjustments |
| `OPENING_BALANCE` | Migration / opening AP |

**Future invariants** (service-enforced, not DB triggers):

```text
outstandingAmount = originalAmount − allocatedAmount − adjustedAmount − writtenOffAmount
(all components >= 0)
```

Vendor invoice posting (Phase 4A4) will create `side = CREDIT`, `documentType = VENDOR_INVOICE`, linked via `sourceVendorInvoiceId` (`@unique` — one open item per invoice).

Status lifecycle: `OPEN` → `PARTIALLY_SETTLED` → `SETTLED` (plus `ON_HOLD`, `DISPUTED`, `REVERSED`, `CANCELLED`).

## Lifecycle enums (schema only in 4A1)

`VendorInvoiceStatus`: `DRAFT` → `PENDING_APPROVAL` → `REJECTED` | `READY_TO_POST` → `POSTED` → `REVERSED` | `CANCELLED`.

Workflow transition services, approval integration, and HTTP routes are **deferred** (4A3+).

## Delete and FK behaviour

| Relationship | On delete |
|--------------|-----------|
| `VendorInvoice` → `VendorInvoiceLine` | **Cascade** |
| `VendorInvoice` → `VendorInvoiceSourceLink` | **Cascade** |
| `PayableOpenItem.sourceVendorInvoiceId` → `VendorInvoice` | **Restrict** (invoice cannot be deleted while open item exists) |
| `MasterVendor` → invoice | No FK — soft ref only |

## Readiness for Phase 4A3 (draft workflow + API)

4A1 + 4A2 provide:

1. Typed header/line columns for gross → tax → grand total → TDS → vendor payable chain.
2. `calculationVersion`, `calculationSnapshot`, `accountingPreviewSnapshot` JSON placeholders.
3. Repositories that accept persisted totals once a workflow service writes them.
4. Normalisation helpers ready for duplicate detection (already live in 4A2's `assessVendorInvoiceDuplicates`) and uniqueness claim in 4A3.
5. A pure/DB-optional calculation and validation engine (`calculateVendorInvoiceSync` / `calculateVendorInvoice`) that a draft workflow service can call before persisting.
6. Live tests: `finance-ap-vendor-invoice-foundation.test.ts` (schema, repos, isolation, delete semantics), `finance-ap-vendor-invoice-calculation.test.ts` (calculation engine, pure sync), `finance-ap-vendor-invoice-duplicate.test.ts` (duplicate detector, live MySQL).

Next phase: draft workflow (submit/approve/reject/ready-to-post) + HTTP API that calls the calculation engine and persists results — still no posting.
